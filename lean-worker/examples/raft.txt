Scope note (important, please review): "Implement Paxos protocol" describes an entire distributed consensus system involving multiple roles (proposers, acceptors, learners), message passing, persistent state, and failure handling. That cannot be expressed as a single method for this pipeline. This specification covers exactly one core, self-contained rule of Paxos: the proposer's value-selection rule used in Phase 2a. The remaining Paxos components — acceptor promise logic (Phase 1b), acceptor accept logic (Phase 2b), learner quorum/chosen-value detection, proposal-number generation, and any networking or state persistence — are explicitly out of scope and are enumerated above as separate specification runs.

Purpose and behavior: After a proposer receives promise responses from a quorum of acceptors, Paxos requires it to propose the value associated with the highest-numbered previously accepted proposal among those responses; only if no acceptor in the quorum has previously accepted any value may the proposer use its own value. This method implements that selection rule as a pure function.

Inputs:
- acceptedNums: Array Nat — the proposal numbers of previously accepted proposals reported in the promise responses. Each entry corresponds positionally to the entry at the same index in acceptedVals. Only responses that actually reported a previously accepted proposal are included; responses reporting "nothing accepted yet" must be filtered out by the caller before invoking this method.
- acceptedVals: Array Int — the values of those previously accepted proposals, positionally aligned with acceptedNums.
- defaultVal: Int — the proposer's own desired value, used only when no acceptor reported a previously accepted proposal.

Output:
- result: Int — the value the proposer must propose in Phase 2a.

Preconditions (explicit):
- acceptedNums.size = acceptedVals.size. Behavior is unspecified if the lengths differ; callers must guarantee equal lengths.
- Proposal numbers are natural numbers; there is no upper bound.

Behavior specification:
- If both arrays are empty (no acceptor reported a prior acceptance), the result is defaultVal.
- Otherwise, the result is acceptedVals[i] where i is an index at which acceptedNums attains its maximum value.
- Tie-breaking assumption: in standard Paxos, distinct acceptors reporting the same proposal number must have accepted the same value (this is a protocol invariant), so ties are semantically harmless. For determinism at the specification level, we require that if the maximum proposal number occurs at multiple indices, the value at the FIRST (lowest) such index is returned. This is a recorded assumption a reviewer may change.

Boundaries: The maximum is inclusive; an array of size 1 trivially selects its only value. Proposal number 0 is a legal accepted proposal number (assumption: the caller does not reserve 0 as a sentinel; reviewer may change this).

Mutation and ordering: The method does not mutate its inputs. Input ordering matters only for the first-occurrence tie-break; otherwise the result is order-insensitive.

Determinism: Fully deterministic — identical inputs always produce identical output.

Edge and degenerate cases:
- Empty inputs: returns defaultVal.
- Single element: returns that element's value regardless of defaultVal.
- All proposal numbers equal: returns the value at index 0.
- defaultVal is ignored whenever the arrays are non-empty, even if defaultVal is "better" by any measure.

Complexity requirement: O(n) time in the length of the arrays, O(1) additional space.

Worked example: acceptedNums = [3, 7, 5], acceptedVals = [10, 20, 30], defaultVal = 0. The maximum proposal number is 7 at index 1, so the result is 20.

Second worked example: acceptedNums = [], acceptedVals = [], defaultVal = 42. No prior acceptances exist, so the result is 42.

=== METHOD_SIGNATURE ===

method chooseProposalValue(acceptedNums: Array Nat, acceptedVals: Array Int, defaultVal: Int) returns (result: Int)
